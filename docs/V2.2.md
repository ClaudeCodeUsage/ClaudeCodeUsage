# V2.2 — Polish, share, and maintain

V2.2 makes the extension feel more polished, more shareable, and easier to
maintain, without changing the core identity: a clean Claude Code usage
dashboard, a quota/status helper, and a usage coach. It is a **patch-and-feature
release line** — small, reviewable changes over flashy rewrites.

Status legend: ✅ implemented (with tests) · 🟡 logic done, UI integration is a
follow-up PR · 📝 planned / spec only.

## 2. Usage Share Card / 用量卡片

A purpose-designed, attractive social-share card — **not** a screenshot of the
dashboard. Visual direction `Code Pulse`: clean modern card, soft CSS gradient,
subtle CSS-only texture, matte inner panels, rounded corners, generous
whitespace, one bold hero metric, compact supporting metrics, one rhythm visual,
one playful badge, a small footer watermark. Two real compositions: **landscape
1600×900** (X / LinkedIn / GitHub) and **portrait 1080×1440** (小红书 / 微信) —
the portrait is a true portrait layout, not a stretched landscape.

- **Command:** `Claude Code Usage: Export Usage Share Card`; **Share** buttons on
  the Today and Month pages.
- **Ranges:** Today / This week / This month (custom range is a documented TODO).
- **Themes:** follow VS Code / Light / Dark — all readable.
- **Selectable sections** (`ShareSections`, see `shareCard.ts`): default ON —
  total tokens, estimated cost, sessions, top model, cache efficiency, rhythm,
  badge, watermark. Default OFF — project name, workflow share, peak context,
  detailed token composition.
- **Badges** (local, deterministic — `selectShareBadge`): Context Marathoner,
  Cache Saver, Token Sprinter, Workflow Pilot, Steady Builder.

### Privacy model (enforced, not just promised)
`ShareCardData` only ever carries aggregate, non-identifying fields — there is
**no key** in the type for prompt text, file paths, session ids, usernames, or
absolute directories, so they cannot leak by accident (a test asserts this).
Project name is **hidden by default**; enabling it shows a short privacy hint.
Nothing is uploaded — export is a local PNG via the VS Code save dialog
(`shareCardFilename`, e.g. `claude-code-usage-2026-06-week.png`).

### Rendering / export (next PR)
Render the card as a fixed-size DOM node inside the existing webview and rasterise
with a mature, lightweight DOM-to-image library — **`html-to-image`** is the
candidate (tiny, no native deps, MIT, pure-DOM). **Dependency justification:**
hand-rolling SVG/`foreignObject` rasterisation in a CSP-restricted webview is the
classic fragile path; `html-to-image` is the smallest well-maintained option.
Constraints: no remote images / fonts / CSS; inline all icons/decorations; do
**not** weaken the webview CSP (it currently allows only `'unsafe-inline'`
style/script — image rasterisation stays within the webview, no `connect-src`
needed). If the dependency proves incompatible with the CSP, fall back to an
SVG-string export.

## 3. Token heatmap (Monthly tab)

A GitHub-contribution-inspired month grid (`src/heatmap.ts`, tested): one rounded
cell per calendar day, intensity by tokens (default) / cost / sessions, zero-fill
for empty days, 0–4 buckets by quartile of the month's max. Per-day tooltip:
date, tokens, est. cost, sessions. **No GitHub integration, no OAuth, no upload**
in V2.2. Theme-variable colours (a clean low-saturation scale, not GitHub green),
`Less`/`More` legend, compact so it enriches the Monthly tab without crowding the
existing charts. The share card's rhythm strip reuses the same daily aggregation.

## 5. Session / quota improvements

- **Quota resilience** — already in 2.1 and preserved: the last successful
  `/usage` result is cached; TTL throttles refetch to ≥ 60 s (well past the 30 s
  floor); a 429 triggers a 60 s cool-down; failures degrade gracefully (the item
  hides, the status bar is never broken). 📝 *Remaining:* a small "fresh vs
  cached" hint in the quota tooltip.
- 📝 **Sessions tab** (design — next PR, kept minimal): per-row **Copy session
  id** and **Copy project path**; **Resume session** (same project → prefer the
  official Claude Code mechanism if present; otherwise a terminal `claude
  --resume <id>`); a **filter** (Current project / All, default Current); the
  active tab already persists across refresh. **Delete session** is deliberately
  deferred — safe cross-platform move-to-trash is non-trivial; it ships only with
  a confirmation modal + OS-trash (never a hard delete), or stays a documented
  future item.
- 📝 **Detail width** — allow ~1600px / responsive on wide screens without
  looking stretched on normal ones.

## 6. Contributor & PR credit policy

- Adopted issues and small PRs are acknowledged in the README on release.
- A high-value small PR gets credit **even if** its work is later folded into a
  larger maintainer PR.
- When a PR is closed because it was integrated elsewhere, leave a clear comment
  saying where it shipped — and prefer **squash-merge** (which preserves Git
  attribution and the "merged" status) over close-and-reapply.
- PRs that resolve a tracked issue use `Closes #N` so the issue auto-closes with
  a link on merge.

## 7. Planning (not implemented in V2.2)

### 7.1 AI advice enhancement
📝 Detect when a session has grown too long or shifted topic (large context
share, many turns, a clear topic change in the prompt sample) and suggest
starting a **new session** or running **`/clear`**. Deterministic signals first;
the advice model phrases it.

### 7.2 Usage Optimizer — local file reading (opt-in)
📝 Default: **no local file reads** (today's behaviour). Future opt-in levels,
each strictly additive and consented:
(a) prompt files only → (b) docs files → (c) code files. Always **show exactly
what will be read** before sending; **never** read hidden/secret files
(`.env`, `.git`, credentials, dotfiles); **never** upload content without an
explicit user action.

### 7.3 Local conversation export
📝 Export a conversation by project / session, distinguishing user / assistant /
tool records where possible, with default privacy redaction. Local file only.

### 7.4 Subscription-durability sharing

precision; note that enterprise/team plans may not be comparable.
