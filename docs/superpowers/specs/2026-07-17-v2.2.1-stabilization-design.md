# V2.2.1 Stabilization Design

**Date:** 2026-07-17
**Status:** Approved by the maintainer on 2026-07-17
**Target:** `codex/v2.2.1-stabilization`, based on `upstream/main` at `v2.2.0`
**Chinese review copy:** [简体中文版](2026-07-17-v2.2.1-stabilization-design.zh-CN.md)

## Objective

Ship a focused v2.2.1 that restores a trustworthy development baseline, reduces
the high-CPU refresh path reported in issue #70, lands the already-reviewed
small bug fixes, and makes Codex-assisted maintenance attribution explicit.
The release must preserve Claude usage totals and the project's local-first,
read-mostly trust model.

## Constraints

- Keep the product Claude-only in v2.2.1. The Codex usage adapter starts after
  this release.
- Add no runtime dependency.
- Exact usage totals and estimated insights remain separate.
- Never log prompt content, absolute paths, session IDs, credentials, or raw
  JSONL lines in performance diagnostics.
- New settings preserve current behavior by default. Existing users are not
  silently migrated to a different refresh cadence.
- All user-facing strings go through all seven UI locales.
- README changes update the six current README files together; Indonesian is
  added to that rule only if PR #76 lands.
- Any artifact that needs maintainer review—including a spec, implementation
  plan, design, release checklist, policy, or substantial documentation
  change—has a faithful Simplified-Chinese companion or review note, and the
  Chinese link is presented first.
- Every behavior change follows a red-green TDD cycle and the full compiled
  `node:test` suite must pass before release.

## Baseline and evidence

The release baseline is the clean v2.2.0 tree, not the old local feature
branch. On that baseline, compile succeeds and 114 tests pass.

The v2.2.0 refresh path has three independently verified costs:

1. A JSONL watcher event marks the extension active for 60 seconds. During
   that window, polling is forced to 8 seconds and ignores the configured
   30–3600 second `refreshInterval`.
2. Every poll recursively enumerates and stats all JSONL files. When any file
   changed, the loader enumerates again, reads every file once to discover its
   earliest timestamp, and reads every file again to parse usage.
3. The timestamp pass uses unbounded `Promise.all`, creating avoidable decode,
   allocation, and garbage-collection pressure.

The maintainer's current corpus has 389 JSONL files totalling about 532 MiB.
A changing refresh therefore requests roughly 1.04 GiB of logical JSONL reads.
This proves that the path is hazardous, but it does not prove that it is the
only hotspot on the Linux reporter's machine; the issue still needs a CPU
profile and prerelease validation.

## Release structure

V2.2.1 is delivered as focused, independently reviewable changes.

### Change 0: Codex takeover and packaging safety

- Replace the inaccurate root `AGENTS.md` with instructions derived from the
  v2.2 architecture and contribution rules.
- Keep the product identity as Claude Code Usage; document Codex as a
  development tool, not a completed provider integration.
- Restore the v2.2 `.gitignore` and `.vscodeignore` protections for private
  documents, local guidance, secrets, tests, and GitHub workflow sources.
- Add line-ending and file-mode guidance so a future add does not recreate the
  current CRLF/executable-bit contamination.
- Record that push, PR, merge, and release operations require maintainer
  approval.

### Change A: Low-risk CPU mitigation and diagnostics

Polling and watch semantics are separated:

- `refreshInterval` always controls fallback polling. Active usage no longer
  silently replaces it with an 8-second interval.
- `fileWatchSeconds` means quiet debounce only: refresh after no JSONL write has
  arrived for the selected duration.
- Add 60, 120, and 300 second live-refresh choices. Keep the existing 2-second
  default to avoid an unannounced behavior migration.
- Continuous writes coalesce into one pending refresh rather than creating a
  second immediate full scan after every in-flight scan.
- Manual refresh remains an explicit full reconciliation.

The diagnostic log gains anonymous stage measurements:

- trigger: startup, watch, poll, focus, settings, or manual;
- files discovered, changed, reused, and removed;
- bytes read and lines parsed;
- manifest, read/parse, aggregate/render, and total duration;
- watcher event and coalescing counts.

`getEarliestTimestamp` reads only enough of the file to find the first valid
timestamp and uses at most eight concurrent readers. It must never read the
entire corpus solely to sort files.

Until v2.2.1 is installed, the documented workaround for #70 is:

1. Set Live refresh delay to Off.
2. Set Refresh interval to 300–900 seconds.
3. Optionally turn Content analysis off.

Turning Dashboard auto-refresh off alone is not advertised as a CPU fix,
because status-bar parsing still runs.

### Change B: In-memory per-file usage index

The root fix is an in-memory `ClaudeUsageIndex` owned by the extension. It uses
a manifest entry containing path, size, modification time, device, and inode
when available.

- Cold start reads each file once.
- An unchanged file reuses its parsed contribution.
- A changed or new file is re-read in full in v2.2.1. Byte-offset tail parsing
  is deliberately deferred until the simpler cache is proven correct.
- Deleted files remove their previous contribution.
- Truncation, inode change, or atomic replacement invalidates the file entry.
- Watch filenames are hints only; manifest reconciliation is authoritative.
- Cache data stays in memory and is never persisted.
- Content-analysis window, parsing contract, and attribution-rule changes have
  explicit cache-version invalidators.
- Startup, manual refresh, and a 30-minute periodic reconciliation recover from
  dropped watcher events.

The index produces the same normalized records as a cold full load. Cross-file
deduplication, session titles, user-message counting, workflow/subagent
attribution, and content-analysis calibration remain global aggregation steps.

If Change B does not pass cold-versus-incremental equality tests in time,
v2.2.1 may ship Change A as a mitigation. In that case issue #70 remains open
and the release notes must not claim that the root cause is resolved.

## Codex attribution and automation boundary

Codex-authored GitHub text ends with:

```md
---
🤖 Generated with [OpenAI Codex](https://developers.openai.com/codex/)
```

An unreviewed automated first pass uses the stronger wording:

```md
---
🤖 Generated by [OpenAI Codex](https://developers.openai.com/codex/) as an automated first pass — not a maintainer decision.
```

Attribution must describe the actual generator. The current first-pass script
uses an Anthropic-compatible API with DeepSeek as its default, so it must not
be relabelled Codex without changing the runner. The wrapper, not the model
prompt, deterministically adds exactly one trusted footer.

V2.2.1 documents the distinction between:

- maintainer-reviewed Codex-assisted comments;
- controlled, comment-only first-pass automation; and
- the privileged maintainer-only mention agent that can write repository
  content.

Codex/OpenAI receives development-tool credit in all README language versions.
It is not added to Release Drafter's human contributor list and no fabricated
`Co-Authored-By` identity is used.

## Verification and release gate

Automated tests cover:

- cadence always respecting `refreshInterval`;
- watcher burst coalescing and the new debounce choices;
- bounded first-timestamp reads;
- cold load, unchanged refresh, one changed file, new/delete/truncate/replace;
- an incomplete final JSONL line completed on the next refresh;
- global placeholder/real-record deduplication and late title updates;
- workflow and subagent attribution equality;
- content-analysis invalidation;
- deep equality between incremental output and a fresh cold load;
- deterministic, provider-correct GitHub attribution footers.

Release evidence requires:

- TypeScript compile exit 0;
- full `node:test` suite with zero failures;
- a packaged VSIX whose file list excludes private docs, local instructions,
  workflow source, and test output;
- F5/installed-VSIX smoke tests on macOS and Linux when available;
- a prerelease VSIX offered to the #70 reporter with diagnostic instructions.

## Non-goals

- Codex usage ingestion or multi-provider UI;
- achievements, telemetry, cloud sync, or automatic memory writes;
- PR #73's 1000-row sessions rendering;
- persistent caching of prompt-bearing data;
- claiming Linux `fs.watch` is the root cause without a profile.
