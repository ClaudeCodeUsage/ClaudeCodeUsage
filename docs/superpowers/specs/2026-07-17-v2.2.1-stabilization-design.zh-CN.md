# V2.2.1 稳定化设计

**日期：** 2026-07-17
**状态：** 维护者已于 2026-07-17 批准
**目标：** `codex/v2.2.1-stabilization`，基于 `upstream/main` 的 `v2.2.0`
**英文原版：** [English](2026-07-17-v2.2.1-stabilization-design.md)

## 目标

发布一个聚焦的 v2.2.1：恢复可信的开发基线，降低 issue #70 所报告的高 CPU 刷新路径开销，合入已经审阅的小型缺陷修复，并明确标注由 Codex 辅助完成的维护工作。此版本必须保持 Claude 用量总计不变，并维持项目本地优先、以读取为主的信任模型。

## 约束

- v2.2.1 中产品仍仅支持 Claude。Codex 用量适配器在此版本之后启动开发。
- 不增加运行时依赖。
- 精确用量总计与估算型洞察继续分开处理。
- 性能诊断绝不记录提示词内容、绝对路径、会话 ID、凭据或原始 JSONL 行。
- 新设置默认保持当前行为。不得在未告知的情况下将现有用户迁移到不同的刷新节奏。
- 所有面向用户的字符串都必须覆盖全部七种 UI locale。
- README 变更必须同时更新当前六份 README 文件；只有 PR #76 合入后，才把印度尼西亚语加入这条规则。
- 任何需要维护者审阅的 artifact（包括 spec、implementation plan、design、release checklist、policy 或大幅文档变更）都必须提供忠实的简体中文副本或中文审阅说明，并优先展示中文链接。
- 每项行为变更都遵循红—绿 TDD 循环；发布前必须通过完整的已编译 `node:test` 测试套件。

## 基线与证据

发布基线是干净的 v2.2.0 代码树，而不是旧的本地功能分支。在该基线上，编译成功且 114 项测试全部通过。

v2.2.0 的刷新路径存在三项已独立验证的开销：

1. JSONL watcher 事件会把扩展标记为活跃 60 秒。在此期间，轮询被强制设为 8 秒，并忽略已配置的 30–3600 秒 `refreshInterval`。
2. 每次轮询都会递归枚举所有 JSONL 文件并执行 stat。只要任意文件发生变化，loader 就会再次枚举；先读取每个文件以发现最早时间戳，再次读取每个文件以解析用量。
3. 时间戳读取阶段使用无界 `Promise.all`，造成原本可以避免的解码、内存分配和垃圾回收压力。

维护者当前的数据集包含 389 个 JSONL 文件，总计约 532 MiB。因此，只要发生变化，一次刷新就会请求约 1.04 GiB 的逻辑 JSONL 读取量。这证明该路径具有风险，但不能证明它是 Linux 报告者机器上的唯一热点；该 issue 仍需 CPU profile 和预发布验证。

## 发布结构

V2.2.1 由若干聚焦且可独立审阅的变更组成。

### 变更 0：Codex 接手与打包安全

- 用根据 v2.2 架构和贡献规则整理的指引，替换根目录中不准确的 `AGENTS.md`。
- 保持产品身份为 Claude Code Usage；将 Codex 记录为开发工具，而不是已经完成的 provider 集成。
- 恢复 v2.2 的 `.gitignore` 和 `.vscodeignore` 保护，覆盖私有文档、本地指引、密钥、测试以及 GitHub workflow 源文件。
- 增加换行符和文件模式指引，避免未来执行 add 时重现当前的 CRLF／可执行位污染。
- 记录 push、PR、merge 和 release 操作都需要维护者批准。

### 变更 A：低风险 CPU 缓解与诊断

将轮询语义与 watch 语义分离：

- `refreshInterval` 始终控制兜底轮询。活跃使用状态不再静默地把它替换为 8 秒间隔。
- `fileWatchSeconds` 只表示静默 debounce：在所选时长内未再发生 JSONL 写入后才刷新。
- 增加 60、120 和 300 秒的实时刷新选项。保留现有 2 秒默认值，避免未经说明的行为迁移。
- 连续写入合并为一次待执行刷新，而不是在每次进行中的扫描结束后再立即触发第二次完整扫描。
- 手动刷新仍是显式的完整 reconciliation。

诊断日志增加匿名阶段测量：

- 触发源：startup、watch、poll、focus、settings 或 manual；
- 已发现、已变更、已复用和已删除的文件数量；
- 读取的字节数与解析的行数；
- manifest、read/parse、aggregate/render 以及总耗时；
- watcher 事件数和合并次数。

`getEarliestTimestamp` 只读取足以找到首个有效时间戳的文件内容，并且最多使用八个并发 reader。绝不能仅为了排序文件而读取整个数据集。

在安装 v2.2.1 之前，#70 的文档化临时解决方案为：

1. 将 Live refresh delay 设置为 Off。
2. 将 Refresh interval 设置为 300–900 秒。
3. 可选：关闭 Content analysis。

不应仅把关闭 Dashboard auto-refresh 宣传为 CPU 修复，因为状态栏解析仍会运行。

### 变更 B：内存中的逐文件用量索引

根本修复方案是由扩展持有的内存内 `ClaudeUsageIndex`。它使用包含 path、size、modification time，以及在可用时包含 device 和 inode 的 manifest entry。

- 冷启动时每个文件只读取一次。
- 未发生变化的文件复用其已解析贡献值。
- 在 v2.2.1 中，发生变化或新建的文件会被完整重读。基于字节偏移量的尾部解析有意推迟，待这个更简单的缓存经验证正确后再实施。
- 文件被删除时，移除其先前的贡献值。
- 文件截断、inode 变化或原子替换会使相应文件条目失效。
- watch 文件名只作为提示；manifest reconciliation 才是权威来源。
- 缓存数据只保留在内存中，绝不持久化。
- 内容分析窗口、解析契约和归属规则的变化都有显式的 cache-version invalidator。
- 启动、手动刷新，以及每 30 分钟一次的周期性 reconciliation 用于从丢失的 watcher 事件中恢复。

该索引产生的 normalized records 必须与冷启动完整加载一致。跨文件去重、会话标题、用户消息计数、workflow／subagent 归属以及内容分析校准仍作为全局聚合步骤执行。

如果变更 B 无法及时通过 cold-versus-incremental equality tests，v2.2.1 可以仅发布变更 A 作为缓解措施。此时 issue #70 必须保持打开，release notes 也不得声称根因已解决。

### 变更 C：现有 issue 与 PR 队列

合入／审阅顺序为：

## Codex 署名与自动化边界

由 Codex 撰写的 GitHub 文本以以下内容结尾：

```md
---
🤖 Generated with [OpenAI Codex](https://developers.openai.com/codex/)
```

未经审阅的自动化 first pass 使用语气更强的以下表述：

```md
---
🤖 Generated by [OpenAI Codex](https://developers.openai.com/codex/) as an automated first pass — not a maintainer decision.
```

署名必须描述实际生成器。当前 first-pass 脚本使用兼容 Anthropic 的 API，并以 DeepSeek 为默认值，因此在不更换 runner 的情况下不得把它重新标记为 Codex。可信 footer 必须由 wrapper 而不是模型 prompt 以确定性方式添加，并确保恰好出现一次。

V2.2.1 记录以下三者的区别：

- 经维护者审阅、由 Codex 辅助完成的评论；
- 受控且只能发表评论的 first-pass 自动化；
- 仅限维护者使用、可写入仓库内容的高权限 mention agent。

Codex/OpenAI 在所有 README 语言版本中获得开发工具署名。它不加入 Release Drafter 的人类贡献者名单，也不使用虚构的 `Co-Authored-By` 身份。

## 验证与发布门槛

自动化测试覆盖：

- 刷新节奏始终遵守 `refreshInterval`；
- watcher burst 合并与新增 debounce 选项；
- 有界的首个时间戳读取；
- 冷加载、未变化刷新、单文件变化、新建／删除／截断／替换；
- 不完整的最后一行 JSONL 在下次刷新时补全；
- 全局 placeholder／真实记录去重以及延迟标题更新；
- workflow 和 subagent 归属一致性；
- 内容分析失效处理；
- 增量输出与全新冷加载之间的深度相等；
- 确定性且 provider 正确的 GitHub 署名 footer。

发布证据要求：

- TypeScript 编译退出码为 0；
- 完整 `node:test` 测试套件零失败；
- 打包后的 VSIX 文件列表不包含私有文档、本地指引、workflow 源文件和测试输出；
- 在可用时，分别在 macOS 和 Linux 上完成 F5／已安装 VSIX 冒烟测试；
- 向 #70 报告者提供预发布 VSIX 以及诊断说明。

## 非目标

- Codex 用量摄取或多 provider UI；
- 成就、遥测、云同步或自动写入 memory；
- PR #73 的 1000 行 sessions 渲染；
- 对含提示词数据进行持久缓存；
- 在缺少 profile 的情况下声称 Linux `fs.watch` 是根因。
